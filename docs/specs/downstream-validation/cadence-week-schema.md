# Cadence-week report schema

> **Status:** schema doc — explains the artifact that
> [`scripts/changelog_cadence.py`](../../../scripts/changelog_cadence.py)
> emits and that W4.2's hard gate reads. Companion to
> [`tasks.md`](tasks.md) W0.6 (script), W0.7 (workflow), W4.2 (gate).

## What it is

A `cadence-week-{N}.md` file written **once per week** by
[`.github/workflows/cadence-report.yml`](../../../.github/workflows/cadence-report.yml)
(cron `0 12 * * 1` — Mondays noon UTC), under
`docs/specs/downstream-validation/`. Four of them
(`cadence-week-1.md` through `cadence-week-4.md`) together gate
the 0.2.0 cut at W4.2.

## File format

The script emits a fully-deterministic markdown report:

```markdown
# Cadence report: 2026-05-19 → 2026-05-25

> Auto-generated by `scripts/changelog_cadence.py`.
> `Added > 0` resets the Phase 4 freeze clock; status is derived from the count under `Added`.

| Category | Count |
|---|---:|
| `Added` | 0 |
| `Changed` | 17  (releases: 0.1.22, 0.1.21, 0.1.23) |
| `Fixed` | 3 |
| `Security` | 3 |

**Status:** ON TRACK (freeze active, week 1/4)
```

| Field | What it means |
|---|---|
| Heading window | Inclusive `[window_start, window_end]` 7-day range covering one cadence week |
| `Added` | Top-level bullets under `### Added` in `[Unreleased]` plus any dated `## [X.Y.Z] - YYYY-MM-DD` blocks whose date falls in the window |
| `Changed` / `Fixed` / `Security` | Same counting rule; `(releases: ...)` annotation lists which dated releases contributed |
| `Status` | `ON TRACK` iff `Added == 0`, else `RESET` |
| `(freeze active, week N/T)` | Cosmetic — passed via `--freeze-week` / `--freeze-total` |

## What "clean" means

> A cadence week is **clean** iff its emitted status reads `ON TRACK`.

The 4-week burn-in graduates at W4.2 iff `cadence-week-{1,2,3,4}.md`
all read `ON TRACK`. If any week reads `RESET`, the clock resets to
week 0 and the freeze extends by 4 more weeks from the resetting
release.

## How the human gate (W4.2) reads four reports

W4.2 is a hard gate (per [tasks.md](tasks.md) line 53). The
verification at W4 close:

1. List `docs/specs/downstream-validation/cadence-week-*.md`.
2. Confirm exactly four files (`-1` through `-4`).
3. Confirm each ends with `**Status:** ON TRACK ...`.
4. Confirm windows are contiguous (week N+1's `window_start` ==
   week N's `window_end` + 1 day).
5. Confirm the four windows cover the most-recent four cadence
   weeks (no stale leftovers from a prior reset).

If all five hold, W4.2 graduates and W4.3 writes
`exit-summary.md` recommending the 0.2.0 cut.

## Override mechanism (known gap — see below)

Per [tasks.md](tasks.md) §"Freeze semantics" + the
[CHANGELOG.md](../../../CHANGELOG.md) 0.1.22 block, **a `### Added`
entry may ship under a `freeze-override` rationale** when:

- the addition is a quality knob with `Security`-scoped-comparable
  reversibility (e.g. flipping a class-attribute default restores
  pre-change behavior exactly), AND
- the PR carries the `freeze-override` label, AND
- the CHANGELOG section opens with a `> **Freeze override (...)**`
  blockquote containing an `[Override-rationale]` paragraph.

When all three conditions hold, the cadence clock **does not reset**
even though `Added > 0` for that window.

### Known gap (2026-05-22)

`scripts/changelog_cadence.py` **does not currently detect override
blockquotes** — it counts raw `### Added` bullets and emits `RESET`
on any `Added > 0`. A live dry-run for week-1 window
(2026-05-19 → 2026-05-25) currently returns `RESET` because 0.1.22's
`MIN_ALIAS_OVERLAP` Added entry — explicitly authorized under the
override mechanism — is counted as a violation.

**Resolution path:** extend `changelog_cadence.py` to:

1. Parse each `## [X.Y.Z]` block for a leading `> **Freeze override`
   blockquote in the first ~10 lines of the section.
2. If found, subtract that section's Added count from the aggregate.
3. Annotate the report: `| Added | 1 (1 under override) |`.
4. Keep `Status: ON TRACK` when **all** Added entries are
   override-flagged.

This is the W1.1 follow-up — needs to land before the first
auto-fired report on Monday 2026-05-25 reads `RESET` against an
already-authorized addition, which would create false-positive
churn on the watcher and on attune-gui downstream signal.

## Worked examples

### Clean week (W1 if override fix lands)

```
Category counts:  Added=0, Changed=17, Fixed=3, Security=3
Status:           ON TRACK (freeze active, week 1/4)
Verdict:          clock advances; cadence-week-1.md commits
```

### Reset week (raw script today)

```
Category counts:  Added=1, Changed=17, Fixed=3, Security=3
Status:           RESET (freeze active, week 1/4)
Verdict:          if no override-flag detection, clock resets to 0;
                  freeze extends 4 weeks from the resetting release
```

### Override-flagged week (target behavior post-fix)

```
Category counts:  Added=1 (1 under override), Changed=17, Fixed=3, Security=3
Status:           ON TRACK (freeze active, week N/4)
Verdict:          clock advances; override rationale recorded in
                  exit-summary.md for the 0.2.0 cut
```

## File lifecycle

| When | What happens |
|---|---|
| Mondays 12:00 UTC | cron fires `cadence-report.yml`; commits `cadence-week-{N}.md` |
| Per-PR (anytime) | `freeze.yml` blocks new `### Added` entries unless the `freeze-override` label is set + `[Override-rationale]` is in PR body |
| End of W4 | W4.2 reads all four files; gate graduates iff all four `ON TRACK` |
| 0.2.0 cut | files become immutable historical record under `docs/specs/downstream-validation/` |
